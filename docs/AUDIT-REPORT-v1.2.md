# تقرير التدقيق الشامل — Vitesse Eco v1.2

**التاريخ:** 2026-04-16
**النطاق:** تدقيق كامل للكود والمحاسبة وسير العمل وسلامة البيانات
**المراجع:** 3 وكلاء متوازيين فحصوا ~15,000 سطر كود

## الملخص التنفيذي (Executive Summary)

### هل النظام المحاسبي سليم؟

**النظام سليم بشكل عام مع 4 أخطاء عالية الخطورة يجب إصلاحها فوراً.** لا توجد أخطاء حرجة (Critical) — أي لا يوجد خطأ يسبب خسارة مالية فورية في الاستخدام العادي. لكن هناك مسارات قديمة (legacy paths) يمكن أن تُفسد البيانات إذا استُخدمت.

### أهم الأخطاء (4 عالية الخطورة):

1. **A-1 [HIGH]**: `updateSale` لا يقرأ `paid_amount` من قاعدة البيانات — عند تعديل المدير لطلب مؤكد، يُصفَّر المبلغ المدفوع ويعود الطلب كأنه غير مدفوع.
2. **B-2 [HIGH]**: نقطة الدفع القديمة `addPayment` تقبل مبالغ تتجاوز الرصيد المتبقي بدون فحص.
3. **B-3 [HIGH]**: `addPayment` لا تُحدّث `sales.paid_amount` أو `remaining` — الدفعات عبر هذا المسار لا تنعكس على حساب الديون.
4. **E-1 [HIGH]**: `POST /api/profit-distributions` لا يستخدم تحقق Zod — جسم طلب مشوّه يسبب خطأ runtime.

### أخطر المخاطر:

- **سير عمل التوصيل → الدفع → الفاتورة**: صحيح ومحمي بالمعاملات (transactions) بشكل جيد.
- **سير عمل الإلغاء**: ممتاز — 12 خطوة في معاملة واحدة مع حماية ضد التكرار.
- **سير عمل توزيع الأرباح**: يعمل لكن فيه ثغرة في القفل — فترات متداخلة (وليست متطابقة) يمكن أن تتجاوز الحد.
- **سير عمل التسوية**: يعمل لكن فحص الحد الأقصى في مسار الكتابة لا يخصم ديون الاسترداد.

### ما يجب إصلاحه أولاً:
1. إصلاح A-1 (SELECT ناقص في updateSale) — **دقائق**
2. إصلاح B-2/B-3 (توجيه addPayment عبر applyCollection) — **ساعة**
3. إضافة Zod schema لـ profit-distributions POST — **30 دقيقة**
4. إضافة Zod لـ purchases PUT و expenses PUT — **30 دقيقة**

## 8. الأخطاء والقواعد المكسورة وفشل سير العمل

### أخطاء عالية الخطورة (HIGH)

#### BUG-001: updateSale لا يقرأ paid_amount — يصفّر الدفعات عند التعديل
- **الخطورة**: عالية
- **الحالة**: مؤكد من الكود
- **الملف**: `lib/db.js:2797` (SELECT) → `lib/db.js:2856` (استخدام)
- **الوصف**: SELECT في updateSale يجلب `item, quantity, status, cost_price` فقط — بدون `paid_amount`. عند الوصول لـ `oldRows[0].paid_amount` يكون `undefined` → `parseFloat(undefined) || 0` = **0**.
- **الأثر**: عند تعديل المدير لطلب مؤكد (سعر/خصم)، المبلغ المدفوع يُصفَّر، الطلب يظهر كغير مدفوع بالكامل، ديون العميل تتضخم.
- **الإصلاح**: إضافة `paid_amount, payment_status` إلى SELECT في السطر 2797.

#### BUG-002: addPayment تقبل مبالغ تتجاوز الرصيد المتبقي
- **الخطورة**: عالية
- **الحالة**: مؤكد من الكود
- **الملف**: `lib/db.js:1245-1252` + `app/api/payments/route.js:35-48`
- **الوصف**: `addPayment` تُدخل دفعة بدون فحص إذا المبلغ > الرصيد المتبقي. المسار الحديث (`applyCollection`) يفحص بشكل صحيح.
- **الأثر**: يمكن تسجيل دفعة 500€ على طلب رصيده المتبقي 100€.
- **الإصلاح**: توجيه جميع الدفعات عبر `applyCollection` أو إضافة فحص تجاوز.

#### BUG-003: addPayment لا تُحدّث sales.paid_amount/remaining
- **الخطورة**: عالية
- **الحالة**: مؤكد من الكود
- **الملف**: `lib/db.js:1245-1252`
- **الوصف**: على عكس `applyCollectionInTx` التي تُحدّث المبلغ المدفوع والمتبقي وحالة الدفع، `addPayment` تُدخل في جدول `payments` فقط بدون تحديث جدول `sales`.
- **الأثر**: الدفعات عبر هذا المسار لا تظهر في حساب الديون. العميل يبدو مديوناً رغم الدفع.
- **الإصلاح**: توحيد مسار الدفع — كل الدفعات تمر عبر `applyCollection`.

#### BUG-004: profit-distributions POST بدون تحقق Zod
- **الخطورة**: عالية
- **الحالة**: مؤكد من الكود
- **الملف**: `app/api/profit-distributions/route.js:28-47`
- **الوصف**: الجسم يُمرَّر مباشرة لـ `addProfitDistribution` بدون schema validation. حقل `recipients` إذا كان string بدل array يسبب crash.
- **الأثر**: طلب مشوّه = خطأ runtime (يُلتقط كـ 400 لكن بدون رسالة عربية مفيدة).
- **الإصلاح**: إنشاء `ProfitDistributionSchema` في `lib/schemas.js`.

| شراء → مخزون → دفع مورد | **صحيح مع ملاحظات** | `updatePurchase` يكسر المتوسط المرجح |
| تحصيل عميل (FIFO) | **صحيح** | أقفال FOR UPDATE صحيحة، FIFO من الأقدم |

النظام **قابل للاستخدام الإنتاجي** مع إصلاح الأخطاء الأربعة العالية الخطورة. سير العمل الأساسي (بيع → توصيل → تحصيل → فاتورة) **صحيح ومحمي بالمعاملات**. نظام الإلغاء **ممتاز** بـ12 خطوة ذرية. نظام البونص والتسويات **يعمل** مع ثغرة صغيرة في فحص ديون الاسترداد.
