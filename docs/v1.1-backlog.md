## Section 1 — Deferred from v1.0 scope study (Phase A)

### 1.1 Profit Distribution System (توزيع أرباح) — P1, L

Multi-recipient percentage-based distribution of profit or collected
revenue to admin + manager users. Currently only single-recipient
bonus settlements exist; the `profit_distribution` settlement type is
already defined in the UI but cannot express a group split.

**User context:**
> "EXAMPLE COLLECTED 2000 ADMIN1 25% ADMIN2 15% MANAGER1 20% MANAGER2
> 20% MANAGER3 20% MAKE CALCULATION THIS PERCENTAGE CAN BE SET EACH
> TIME OR FROM USER SETTING"

**Blocking accountant questions (7):**

1. **Base for distribution** — gross collected revenue, net profit
   after costs and bonuses, or custom formula?
2. **Percentage source** — pre-configured per user
   (`users.profit_share` column) or entered per distribution?
3. **Sum constraint** — must equal 100%, or can the company retain a
   share (e.g., 80% team / 20% company)?
4. **SAS tax treatment** — declared as bonuses (payroll with social
   charges) or dividends (annual declaration)?
5. **Reversibility** — can a committed distribution be cancelled?
6. **Frequency** — manual on-demand, or scheduled (monthly/quarterly)?
7. **Multi-currency** — future-proofing for EUR-only vs international?

**Estimated effort:** L (1 week minimum, includes migration + UI + tests)

## Section 2 — Technical debt

### 2.1 Aggregate reporting test category — P1, S

**Lesson learned:** Bug 3 (`totalPaid` double-count) was silent in 540
stress-test operations. The stress test verified transactional
operations (create/confirm/cancel/collect) but not aggregate functions.
Add aggregate-correctness invariants as a new test category:

- `totalPaid <= totalSales` per client (already in clients-aggregate-correctness)
- `totalPaid + remainingDebt == sum of confirmed non-cancelled sale totals`
- Dashboard P&L: `revenue - COGS - bonuses == net profit`
- VAT total: `sum(payments.tva_amount WHERE type='collection')
  == revenue / 6`
- New stress test rule: *"aggregate consistency after N cancellations
  + collections + settlements"*

**Estimated effort:** S (4-6 hours)

### 2.2 Idempotency-key convention for accumulators — P2, M

Client-supplied `Idempotency-Key` HTTP header for `/collect`,
`/settle`, and `/payments` routes. Standard Stripe pattern. Lets the
backend distinguish an intentional "collect 300 twice" from an
accidental double-click, which the current FOR UPDATE + overpay checks
cannot do by design.

**Estimated effort:** M (2-3 days, schema + middleware + UI)

### 2.3 Voice pipeline refactor — P3, L

Replace the prompt-engineered Llama extraction with structured output
via function calling (OpenAI-style tool use). Reduces the current 5
stages to 3 and makes JSON parsing bulletproof.

**Estimated effort:** L (1 week+)

### 2.4 HSTS preload list submission — P2, S

Submit `mohammadnl.vercel.app` to https://hstspreload.org once
long-term stability is confirmed (1 year+ of uninterrupted HTTPS).
Already configured with `preload` directive in the response header.

**Estimated effort:** S (30 minutes)

## Section 9 — Process improvements (v1.0 retrospective)

Lessons learned during v1.0 development that should shape v1.1
development and the pre-delivery checklist.

### 9.1 Aggregate testing

Bug 3 (`totalPaid` double-count) was silent in 540 stress-test
operations because the stress test only verified transactional
operations. The aggregate functions (`getClients`, `/api/summary`)
were never asserted against a known-value ledger. v1.1 stress test
must include aggregate-correctness assertions per Item 2.1.

### 9.2 UX click-through testing

Bug 1 (`/clients/[id]` not found) surfaced only when the user clicked
the "details" button — 100% of client detail page loads were broken,
undetected. Phase 1 UI scenarios were not comprehensive enough. v1.1
pre-delivery checklist must include:

- Click-through of every "details" / "view" / "open" button on every
  list page
- Empty-state testing on each page (no data + filtered to zero)
- Permission-boundary testing per role per action (admin, manager,
  seller, driver) — the matrix should be a real spreadsheet, not a
  spot check

### 9.3 Pre-delivery checklist additions

Add to the standard checklist for v1.1+:

- [ ] User has navigated to every linked page from main menu
- [ ] Aggregate values cross-checked against the sales ledger
- [ ] All view / open buttons tested as authenticated users of each role
- [ ] Filter combinations tested (no results, all results, partial)
- [ ] Click-through of every actions column button

### 9.4 HAR file analysis

Encouraging the user to share HAR files when reporting *"didn't work"*
issues was decisive in catching Bugs 1, 2, and 3 in the Phase B cycle.
Make HAR submission a **standard part of v1.1 issue reporting** — put
it in the bug-report template.

### 9.5 Stress-test discovery integration

Phase 0.5 stress test **development** surfaced the double-cancel
idempotency vulnerability mid-implementation (not via the test itself,
via reading the code while writing the rules). The "while building,
observe" pattern is valuable. Continue applying it for v1.1 — don't
wait for the test run to find issues; reading the code with
"what if this runs twice?" in mind catches bugs earlier.
