# تقرير النظام الصوتي — Vitesse Eco v1.2

**التاريخ:** 2026-04-16
**النطاق:** تدقيق جنائي كامل للنظام الصوتي — البنية، تدفق البيانات، الأمان، نقاط الضعف
**التقييم:** النظام **قابل للإنتاج** مع ثغرات توثيقية ومراقبة

---

## 1. الحكم النهائي

### هل النظام يعمل؟
**نعم — النظام يعمل بشكل كامل.** خط أنابيب من 14 مرحلة يحوّل التسجيل الصوتي العربي إلى عمليات تجارية مهيكلة (بيع، شراء، مصروف) مع حماية متعددة الطبقات ضد الهلوسة.

### هل النظام آمن؟
**جزئياً.** المصادقة والتحقق من المدخلات جيدة، لكن:
- مفتاح `GROQ_API_KEY` غير موثق في `.env.example`
- لا توجد مراقبة أو تتبع أخطاء منظم
- لا يوجد مزود بديل إذا تعطل Groq
- معدل الطلبات محفوظ في الذاكرة (يضيع عند إعادة التشغيل البارد)

### هل النظام يتعلم؟
**نعم — حلقة تعلّم ذاتي متكاملة.** كل تصحيح من المستخدم يُحسّن:
- أنماط الاستخراج (per-user + global)
- أسماء الكيانات البديلة (aliases)
- أولوية المطابقة (frequency boosting)

## 3. جداول قاعدة البيانات

### `voice_logs` — سجل التدقيق
| العمود | النوع | الوصف |
|--------|-------|-------|
| id | SERIAL PK | معرف تلقائي |
| date | TEXT | تاريخ الطلب YYYY-MM-DD |
| username | TEXT | من أجرى الطلب |
| transcript | TEXT | مخرج Whisper الخام |
| normalized_text | TEXT | بعد التطبيع |
| action_type | TEXT | register_sale / register_purchase / register_expense |
| action_id | INTEGER | FK للعملية الناتجة (إن حُفظت) |
| status | TEXT | pending / completed / error |
| created_at | TIMESTAMP | وقت الإنشاء |

**ملاحظة نقدية:** الجدول ينمو بلا حدود — لا توجد سياسة أرشفة أو حذف تلقائي.

### `ai_corrections` — سجل التصحيحات
| العمود | النوع | الوصف |
|--------|-------|-------|
| id | SERIAL PK | |
| date | TEXT | |
| username | TEXT | من صحّح |
| transcript | TEXT | ما قاله المستخدم |
| ai_output | TEXT | ما استخرجه الذكاء الاصطناعي |
| user_correction | TEXT | ما كتبه المستخدم |
| action_type | TEXT | نوع العملية |
| field_name | TEXT | أي حقل صُحّح |

### `ai_patterns` — الأنماط المتعلّمة
| العمود | النوع | الوصف |
|--------|-------|-------|
| id | SERIAL PK | |
| pattern_type | TEXT | نوع العملية |
| spoken_text | TEXT | ما قاله المستخدم |
| correct_value | TEXT | القيمة الصحيحة |
| field_name | TEXT | الحقل المستهدف |
| frequency | INTEGER | عدد مرات الاستخدام |
| username | TEXT | فارغ = عام، محدد = شخصي |

**UNIQUE** على (spoken_text, correct_value, field_name, username)

**ملاحظة:** الأنماط الشخصية (per-user) لها أولوية على العامة في بناء الـ prompt.

### `entity_aliases` — الأسماء البديلة المتعلّمة
| العمود | النوع | الوصف |
|--------|-------|-------|
| id | SERIAL PK | |
| entity_type | TEXT | product / client / supplier |
| entity_id | INTEGER | FK للكيان الأصلي |
| alias | TEXT | الاسم كما نُطق أو تُعلّم |
| normalized_alias | TEXT | مُطبَّع للمطابقة |
| source | TEXT | المصدر: seed / user / ai_correction / speech_correction / auto_history / transcript_word |
| frequency | INTEGER | عدد مرات المطابقة |

**INDEX** على (entity_type, normalized_alias) — بحث O(1).

## 5. الخدمات الخارجية

### Groq API
| الخدمة | النموذج | الاستخدام |
|--------|---------|-----------|
| النسخ الصوتي | `whisper-large-v3` | تحويل الصوت العربي → نص |

## 6. نظام التطبيع (voice-normalizer.js)

### خط أنابيب التحويل (بالترتيب):

| المرحلة | الوصف | مثال |
|---------|-------|------|
| 1 | تطبيع الألف (أ إ آ → ا) | إبراهيم → ابراهيم |
| 2 | إزالة التشكيل | بَعْتُ → بعت |
| 3 | إزالة التطويل | مـــرحبـــا → مرحبا |
| 4 | أرقام منطوقة → أرقام | مية → 100، خمسين → 50 |
| 5 | مضاعفات مركبة | تلاتة آلاف → 3000 |
| 6 | حروف عربية → لاتينية | في → V، سي → C، تي → T |
| 7 | تنظيف المسافات | B  M  W → BMW |

### اللهجات المدعومة:
- **شامية:** مية، تلات، أربع، خمس، ست، سبع، تمن، تسع
- **خليجية:** الف (بدلاً من ألف)
- **فصحى:** مئة، ثلاثة، أربعة...

### تغطية الاختبارات:
**104 اختبار ناجح** — يشمل:
- تحويل الأرقام (43 حالة)
- الحروف اللاتينية (23 حالة)
- حالات الحد (15 حالة)
- التطبيع العام (23 حالة)

## 7. نظام الحماية من الهلوسة

### الطبقة 1: القائمة السوداء (رفض قاطع)

27 عبارة محظورة في 4 فئات:

| الفئة | أمثلة | العدد |
|-------|-------|-------|
| يوتيوب عربي | "اشتركوا في القناة"، "لا تنسوا اللايك" | 7 |
| يوتيوب إنجليزي | "subscribe to the channel"، "like and share" | 6 |
| يوتيوب فرنسي | "abonnez-vous"، "n'oubliez pas" | 3 |
| علامات Whisper | "[موسيقى]"، "[applause]"، "[music]" | 4+ |

**قرار تصميمي حرج:** الفحص يتم على النص **الخام** (قبل التطبيع) لأن المطبّع يحوّل "في" إلى "V"، مما سيُفلت عبارات مثل "اشتركوا في القناة".

### الطبقة 2: تحذير ناعم

- نص > 20 حرف بدون أي فعل عملية (بعت، اشتريت، دفعت...)
- **لا يرفض** — يضيف تحذيراً فقط
- يمسك الكلام الحقيقي الذي لا يتعلق بالتجارة (LLM كان سيختلق عملية منه)

### الطبقة 3: تصنيف الفعل الصريح

- إذا وُجد فعل صريح (بعت، اشتريت، دفعت) → يتجاوز تصنيف LLM
- حتمي 100% — لا يعتمد على الذكاء الاصطناعي

### الطبقة 4: كشف الصمت (VoiceButton)

- RMS < 0.02 طوال التسجيل → "لم أسمع شيئاً"
- مدة < 1.5 ثانية → "التسجيل قصير جداً"
- يمنع Whisper من هلوسة نص من تسجيل صامت

```
├──────────────────────────────────┤

└──────────────────────────────────┘
```

### صيغة النتيجة النهائية:
```
totalScore = (fuzzy × 0.4) + (jaro × 0.35) + (context × 0.25) + freqBoost
```

### قواعد القرار:
| النتيجة | القرار | الإجراء |
|---------|--------|---------|
| > 0.6 | مطابقة مؤكدة | استخدام الكيان |
| 0.35 - 0.6 | غامض | عرض 3 مرشحين — المستخدم يختار |
| < 0.35 | غير موجود | وسم ككيان جديد |

**مبدأ أساسي:** لا يختار تلقائياً أبداً في حالة الغموض — المستخدم دائماً يقرر.

### الأسماء البديلة المزروعة (seedProductAliases):
~35 اسم عربي مسبق:
- "الفيشن" → V20 Pro
- "دوبل باتري" → EB30
- "في عشرين" → V20 Pro
- "سور رون" → Sur-Ron
- ... وغيرها

### ذاكرة Fuse المؤقتة:
- TTL: 5 دقائق لكل نوع كيان
- تُبطَل عند حفظ تصحيح (لتحديث فوري)

## 14. نقاط القوة

1. **حماية متعددة الطبقات ضد الهلوسة** — 4 طبقات من الأول (صمت) إلى الأخير (تصنيف الفعل). نادراً ما يختلق عمليات وهمية.

2. **حلقة تعلّم ذاتي حقيقية** — ليست مزيفة. كل تصحيح يُحسّن الأنماط والأسماء البديلة. التردد يُرقّي الأنماط الأكثر استخداماً.

3. **حل كيانات قوي** — 3 طبقات (فوري + غامض + سياقي). Jaro-Winkler يمسك حالات التحويل الحرفي. لا يختار تلقائياً في الغموض.

4. **دعم لهجات عربية واسع** — شامية، خليجية، فصحى. 104 اختبار ناجح على المطبّع.

5. **بوابات أمان في كل مرحلة** — حجم الملف، المدة، الصمت، القائمة السوداء، الحقول المفقودة، سعر البيع.

6. **تدهور رشيق** — غياب بيانات من قاعدة البيانات لا يُعطّل النظام. التعلّم في الخلفية لا يحجب الاستجابة.

7. **سجل تدقيق كامل** — 4 جداول تتتبع كل شيء: الطلبات، التصحيحات، الأنماط، الأسماء.

## 15. نقاط الضعف

1. **نقطة فشل واحدة** — Groq هو المزود الوحيد. تعطّله يعني توقف الصوت.

2. **غياب المراقبة** — لا مقاييس، لا تنبيهات، لا تسجيل منظم. المشاكل تُكتشف فقط عند شكوى المستخدم.

3. **غياب الاختبارات** — المطبّع له 104 اختبار، لكن المسارات الرئيسية (process, learn) ليس لها اختبار واحد.

4. **صفحة واحدة فقط** — النظام الصوتي متاح فقط في لوحة التحكم. المبيعات والمشتريات يمكن أن تستفيد مباشرة.

5. **معدل الطلبات هش** — محفوظ في الذاكرة، يضيع عند إعادة التشغيل. مقبول حالياً لكن لا يصمد أمام الحمل العالي.

## 17. الخلاصة

النظام الصوتي في Vitesse Eco هو **نظام متقدم ومهندس بعناية** يحوّل الكلام العربي إلى عمليات تجارية مهيكلة. حماية الهلوسة متعددة الطبقات، حلقة التعلّم الذاتي، وحل الكيانات بثلاث طبقات — كلها تعمل بشكل صحيح.

**النقص الرئيسي ليس في المنطق بل في البنية التحتية:** غياب المراقبة، غياب الاختبارات للمسارات، ونقطة فشل واحدة (Groq). هذه مشاكل هندسية وليست منطقية — لا تمنع الإنتاج لكنها تزيد مخاطر التشغيل.

**الحكم:** جاهز للإنتاج مع التوصيات الفورية.
